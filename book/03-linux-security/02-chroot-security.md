# chrootのセキュリティ上の問題点を検証する

chrootは、仮想的にあるディレクトリをルートディレクトリとして、そのディレクトリの外にアクセスできないようにする機能です。これは非常に古くからあり、FTPサーバーの運用などでセキュリティのために使われてきました。

### chrootを試し、その問題点を理解する

以下のコマンドを実行して、chrootを試してみましょう。

```bash
$ sudo tar xf sysroot-debian-bullseye.tar.xz

$ sudo cp /etc/resolv.conf ./sysroot-debian-bullseye/etc/

$ sudo chroot ./sysroot-debian-bullseye/
```

これにより、Debianのルートディレクトリに移動しますが、用意していたファイルは見えません。

```bash
$ cat /etc/debian_version

$ ls /home/

$ apt update
```

プロセス状態を確認するために、マウントしてみます。

```bash
$ mount -t proc proc /proc/

$ ps -ax
```

この例では攻撃ではないものの、`chroot`が分離しているのは主にファイルシステムの見え方だけであり、カーネルやネットワーク名前空間は元の環境と共有されたままであることがわかります。

```bash
$ uname -a

$ ip addr
```

chrootの中でこれらのリソースにアクセスすると、別の`chroot`や親となる環境にも影響を与えてしまいます。これを防ぐために、Dockerなどのコンテナ技術では「**名前空間**」が活用されており、環境の分離が行われています。アプリケーション実行基盤の環境分離は必須の要件であるため、セキュリティ上非常に重要です。

### chrootのセキュリティ上の問題点

1. **ファイルシステムの分離が不完全**：chrootは、ルートディレクトリの変更を行うだけで、他のリソース（プロセス、ネットワーク、デバイスなど）に関しては分離を行いません。そのため、chroot内のプロセスがこれらのリソースにアクセスする可能性があります。
1. **プロセスの分離が不十分**：chroot環境内で実行されるプロセスは、元の環境で実行されているプロセスと同じ名前空間で実行されます。そのため、chroot内のプロセスが他のプロセスに影響を与える可能性があります。例えば、chroot内のプロセスが他のプロセスにシグナルを送信することができる場合があります。
1. **権限昇格攻撃へのリスク**：chroot環境内で特権昇格攻撃が成功した場合、攻撃者はchroot環境の外にアクセスできる可能性があります。このような攻撃は、例えば、chroot環境内で特権昇格を行った後に、ルートディレクトリを変更して元の環境に戻ることで実現できます。
1. **デバイスファイルへのアクセス**：chroot環境内にデバイスファイルが含まれている場合、そのデバイスファイルにアクセスすることができます。これにより、攻撃者がシステムの他の部分に影響を与える可能性があります。
1. **ネットワークリソースへのアクセス**：chroot環境内のプロセスは、元の環境と同じネットワーク名前空間を使用するため、ネットワークリソースにアクセスできます。これにより、攻撃者はネットワークを通じてシステム外部へのアクセスや、他のネットワークリソースへの影響を与える可能性があります。

これらの問題点から、現代のシステムでは、環境の分離にはコンテナ技術（DockerやKubernetesなど）や仮想化技術（VMwareやHyper-Vなど）がより適切とされています。これらの技術は、より厳密な環境分離を実現し、セキュリティ上のリスクを軽減することができます。
