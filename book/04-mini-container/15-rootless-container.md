# rootlessコンテナの作り方

ここまでの実装は，多くの操作をroot権限で実行する前提です．しかし，実用的なコンテナでは，ホストのroot権限をできるだけ使わずにコンテナを動かしたい場面があります．そこで出てくるのがrootlessコンテナです．

rootlessコンテナの中心になるのはUser名前空間です．User名前空間を使うと，コンテナ内ではrootに見えるが，ホスト側では一般ユーザーとして扱われる，という状態を作れます．

## 基本の流れ

rootlessコンテナの考え方は次の通りです．

1. 一般ユーザーのプロセスが`CLONE_NEWUSER`で新しいUser名前空間を作る
1. 親プロセスが`uid_map`と`gid_map`を書く
1. 子プロセスは新しいUser名前空間の中でUID 0として見える
1. そのUser名前空間が所有するMount名前空間などに対してcapabilityを持つ
1. ホスト側では一般ユーザー，またはsubuid/subgidの範囲として扱われる

この仕組みにより，コンテナ内のrootがホスト全体のrootと同じではない状態を作れます．これは大きな利点です．コンテナ内でroot権限を持つプロセスが侵害されても，ホスト側では一般ユーザーの権限に閉じ込められる可能性が高くなるからです．

## 難しい点

rootlessには難しい点もあります．一般ユーザーは通常，ホスト側でvethを作ったり，`iptables`を変更したりできません．そのため，実際のrootlessコンテナでは，`slirp4netns`や`pasta`のようなユーザー空間ネットワークを使ったり，権限を持つ補助プログラムに一部の処理を任せたりします．

ファイルシステムにも注意が必要です．コンテナ内でrootが作ったファイルは，ホスト側ではマッピング先のUID/GIDの所有物として見えます．複数UIDを扱うには，`/etc/subuid`や`/etc/subgid`で割り当てられた範囲を使います．

この章のサンプルは1つのUID/GIDだけを対応付ける最小形です．

```text
container uid 0  ->  host uid 1000
container gid 0  ->  host gid 1000
```

実用的なrootlessコンテナでは，もっと広いID範囲を扱います．

```text
container uid 0      ->  host uid 100000
container uid 1      ->  host uid 100001
container uid 2      ->  host uid 100002
...
```

この範囲を使うことで，コンテナ内の複数ユーザーをホスト側のsubuid範囲へ対応付けられます．

## この本での位置づけ

この本では，rootlessコンテナを完成品として実装するところまでは踏み込みません．しかし，User名前空間，`uid_map`，`gid_map`，`setgroups`を自分で触ることで，rootlessの土台は確認できます．

rootlessコンテナを深く追う場合は，次に見るべきものはネットワークとファイルシステムです．ネットワークではユーザー空間ネットワーク，ファイルシステムではIDマップ付きマウントやoverlayfsの扱いが重要になります．
